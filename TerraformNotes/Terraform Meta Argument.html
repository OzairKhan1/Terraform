Terraform Meta-Arguments
Complete Notes with Examples

What are Meta-Arguments?

Meta-arguments are special arguments in Terraform that can be used within any resource block to change the behavior of that resource. Unlike provider-specific arguments, meta-arguments are built into Terraform itself and apply to all resource types.

Terraform supports the following meta-arguments:

  • depends_on – Explicit dependency management
  • count – Create multiple resource instances
  • for_each – Create instances from a map or set
  • provider – Select a non-default provider configuration
  • lifecycle – Control resource lifecycle behavior

1. depends_on

The depends_on meta-argument is used to explicitly declare a dependency between resources when Terraform cannot automatically infer it. By default, Terraform builds a dependency graph automatically, but sometimes the dependency is hidden (e.g., an application that uses a policy but doesn't reference it directly).

Syntax:

resource "aws_instance" "example" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  depends_on = [
    aws_iam_role_policy.example_policy
  ]
}

Real-World Example:

An EC2 instance that runs an app requiring an S3 bucket to exist first, but the instance configuration doesn't directly reference the bucket:

resource "aws_s3_bucket" "app_data" {
  bucket = "my-app-data-bucket"
}

resource "aws_instance" "app_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  # EC2 instance depends on S3 bucket even though
  # the bucket name is not referenced here
  depends_on = [aws_s3_bucket.app_data]
}
📝 Note: Use depends_on sparingly. Prefer implicit dependencies (referencing attributes) when possible, as they are cleaner and more maintainable.

2. count

The count meta-argument allows you to create multiple instances of a resource from a single resource block. When count is set, each instance has a unique index (count.index) starting from 0.

Syntax:

resource "aws_instance" "server" {
  count         = 3
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name = "Server-${count.index}"
  }
}

Example with Variable:

variable "instance_count" {
  default = 3
}

resource "aws_instance" "web" {
  count         = var.instance_count
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name = "WebServer-${count.index + 1}"
  }
}

# Accessing a specific instance
output "first_instance_ip" {
  value = aws_instance.web[0].public_ip
}

# Accessing all instances
output "all_instance_ips" {
  value = aws_instance.web[*].public_ip
}
📝 Note: count creates resources indexed by integer (0, 1, 2...). Removing an item from the middle forces re-creation of subsequent resources. Use for_each for more stable indexing.

3. for_each

The for_each meta-argument creates multiple instances of a resource based on a map or set of strings. Each instance is identified by a string key instead of a numeric index, making it more flexible and stable than count.

Syntax with Set of Strings:

resource "aws_s3_bucket" "buckets" {
  for_each = toset(["dev", "staging", "prod"])

  bucket = "my-app-${each.key}"

  tags = {
    Environment = each.key
  }
}

Syntax with Map:

resource "aws_instance" "servers" {
  for_each = {
    web = "t2.micro"
    app = "t2.small"
    db  = "t2.medium"
  }

  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = each.value

  tags = {
    Name = each.key
  }
}

# Access a specific instance
output "web_server_ip" {
  value = aws_instance.servers["web"].public_ip
}

for_each with Variable Map:

variable "environments" {
  type = map(object({
    instance_type = string
    region        = string
  }))
  default = {
    dev  = { instance_type = "t2.micro",  region = "us-east-1" }
    prod = { instance_type = "t2.large",  region = "us-west-2" }
  }
}

resource "aws_instance" "env_servers" {
  for_each      = var.environments
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = each.value.instance_type

  tags = {
    Name        = "Server-${each.key}"
    Environment = each.key
  }
}
📝 Note: each.key gives the map key or set value. each.value gives the map value. for_each is preferred over count when resources have distinct identities.

4. provider

The provider meta-argument allows you to select a specific provider configuration when multiple configurations of the same provider exist (e.g., deploying resources to multiple AWS regions or accounts).

Example – Multi-Region Deployment:

# Default provider (us-east-1)
provider "aws" {
  region = "us-east-1"
}

# Alias for another region
provider "aws" {
  alias  = "west"
  region = "us-west-2"
}

# Resource using default provider
resource "aws_instance" "east_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
}

# Resource using aliased provider
resource "aws_instance" "west_server" {
  provider      = aws.west
  ami           = "ami-0abc123def456"
  instance_type = "t2.micro"
}
📝 Note: The provider meta-argument uses the format <provider_name>.<alias>. Without it, Terraform uses the default (non-aliased) provider configuration.

5. lifecycle

The lifecycle meta-argument provides fine-grained control over how Terraform creates, updates, and destroys resources. It is a nested block with several available settings.

5a. create_before_destroy

By default, Terraform destroys the old resource before creating the new one. Setting create_before_destroy = true reverses this — helpful for zero-downtime deployments.

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  lifecycle {
    create_before_destroy = true
  }
}

5b. prevent_destroy

Prevents accidental destruction of critical resources. Terraform will throw an error if a destroy operation is planned for this resource.

resource "aws_db_instance" "production_db" {
  identifier     = "prod-db"
  engine         = "mysql"
  instance_class = "db.t3.medium"
  username       = "admin"
  password       = "supersecret"

  lifecycle {
    prevent_destroy = true
  }
}

5c. ignore_changes

Tells Terraform to ignore changes to specific attributes after initial creation. Useful when external processes modify a resource (e.g., auto-scaling changes instance count).

resource "aws_instance" "app" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name = "AppServer"
  }

  lifecycle {
    # Ignore changes to ami and tags after creation
    ignore_changes = [ami, tags]
  }
}

# Ignore ALL changes (use with caution)
resource "aws_instance" "managed_externally" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  lifecycle {
    ignore_changes = all
  }
}

5d. replace_triggered_by

Forces replacement of a resource when the referenced resource or attribute changes. Available from Terraform v1.2+.

resource "aws_launch_template" "lt" {
  name          = "my-launch-template"
  image_id      = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
}

resource "aws_autoscaling_group" "asg" {
  desired_capacity = 2
  min_size         = 1
  max_size         = 5

  launch_template {
    id      = aws_launch_template.lt.id
    version = "$Latest"
  }

  lifecycle {
    # Replace ASG whenever the launch template changes
    replace_triggered_by = [aws_launch_template.lt]
  }
}
📝 Note: Use prevent_destroy for databases and critical resources, and create_before_destroy for high-availability services to ensure zero downtime.

Quick Reference Summary

Meta-Argument Purpose When to Use
depends_on Declare explicit dependencies When Terraform can't infer the dependency automatically
count Create N identical resources Creating multiple similar resources with a numeric index
for_each Create resources from map/set When each resource needs a unique key-based identity
provider Select provider configuration Multi-region or multi-account deployments
lifecycle Control create/update/destroy Critical resources, zero-downtime, external changes
