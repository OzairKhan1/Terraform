Terraform State Commands
Production Reference Guide with Examples

What is Terraform State?

Terraform uses a state file (terraform.tfstate) to keep track of the real-world infrastructure it manages. This file maps your configuration to actual cloud resources. In production, state is usually stored remotely (S3, Azure Blob, Terraform Cloud) to enable team collaboration and locking.

State commands let you inspect, manipulate, and recover the state file — they are among the most powerful and sensitive operations in Terraform.

📝 Always back up your state file before running any state manipulation command in production. Use terraform state pull > backup.tfstate first.

State Commands

Safe terraform state list

Lists all resources currently tracked in the Terraform state file. The first thing you run before any state operation to understand what Terraform is managing.

Usage:

# List all resources
terraform state list

# Filter by module
terraform state list module.vpc

# Filter by resource type
terraform state list aws_instance.*

Sample Output:

aws_instance.web
aws_s3_bucket.app_data
aws_security_group.allow_http
module.vpc.aws_vpc.main
module.vpc.aws_subnet.public[0]
✅ Production use: Run before any refactoring or state manipulation to get the exact resource addresses you'll be working with.
Safe terraform state show <resource>

Displays the full details of a specific resource stored in state, including all attributes and their current values. Useful for inspecting a resource without logging into the cloud console.

Usage:

# Show a specific resource
terraform state show aws_instance.web

# Show a resource inside a module
terraform state show module.vpc.aws_subnet.public[0]

Sample Output:

# aws_instance.web:
resource "aws_instance" "web" {
    ami                    = "ami-0c55b159cbfafe1f0"
    id                     = "i-0abc123def456"
    instance_type          = "t2.micro"
    private_ip             = "10.0.1.15"
    public_ip              = "54.210.167.99"
    tags                   = {
        "Name" = "WebServer"
    }
}
✅ Production use: Use to verify real resource attributes (IDs, IPs, ARNs) or debug drift between config and reality.
Caution terraform state mv <source> <destination>

Moves a resource from one state address to another. This is essential when renaming resources in code or restructuring into modules — it prevents Terraform from destroying and recreating the actual infrastructure.

Usage:

# Rename a resource
terraform state mv aws_instance.web aws_instance.app_server

# Move into a module
terraform state mv aws_instance.web module.app.aws_instance.web

# Move a resource out of a module
terraform state mv module.app.aws_instance.web aws_instance.web

# Move between two modules
terraform state mv module.old.aws_instance.web module.new.aws_instance.web
📝 After running state mv, run terraform plan to confirm no unexpected destroy/create actions are planned.
Caution terraform state rm <resource>

Removes a resource from the state file without destroying the actual infrastructure. Terraform simply forgets about it. Used when transferring resource management to another team, tool, or Terraform workspace.

Usage:

# Remove a single resource from state
terraform state rm aws_instance.web

# Remove a module and all its resources
terraform state rm module.vpc

# Remove a specific instance from a count-based resource
terraform state rm 'aws_instance.web[2]'
⚠️ After removal, if you run terraform apply without updating your config, Terraform will try to CREATE the resource again. Either remove it from your .tf files or re-import it.
Safe terraform state pull

Downloads and prints the current state file to stdout. Works for both local and remote backends. Used for backups, auditing, and manual inspection of the full state.

Usage:

# View the full raw state
terraform state pull

# Save state as a backup file
terraform state pull > backup_$(date +%Y%m%d_%H%M%S).tfstate

# Pretty-print and inspect specific resource
terraform state pull | jq '.resources[] | select(.name == "web")'
✅ Production use: Always pull and save a backup before any risky state operation. Combine with jq for powerful querying of the JSON state.
Dangerous terraform state push <file>

Uploads a local state file to the configured remote backend, overwriting the current state. Used for state recovery after corruption or to restore a previous backup.

Usage:

# Push a local state file to remote backend
terraform state push backup_20240101.tfstate

# Force push (override serial number check)
terraform state push -force backup_20240101.tfstate
⚠️ DANGER: This overwrites the remote state completely. Only use during disaster recovery and with full team awareness. A wrong push can cause mass resource destruction on the next apply.
Caution terraform state replace-provider <old> <new>

Replaces the provider source recorded in the state for all matching resources. Used when migrating from an unofficial or old provider path to the official registry path.

Usage:

# Migrate from old provider path to new
terraform state replace-provider \
  "registry.terraform.io/-/aws" \
  "registry.terraform.io/hashicorp/aws"

# Migrate a community provider
terraform state replace-provider \
  "hashicorp/kubernetes" \
  "registry.terraform.io/hashicorp/kubernetes"
📝 Run terraform init and terraform plan after this to ensure the new provider is properly initialized and no changes are unintended.
Caution terraform import <address> <real-id>

Brings an existing real-world resource under Terraform management by adding it to the state file. Widely used when adopting Terraform for infrastructure that was created manually or by another tool.

Usage:

# Import an EC2 instance
terraform import aws_instance.web i-0abc123def456789

# Import an S3 bucket
terraform import aws_s3_bucket.app_data my-app-data-bucket

# Import a resource inside a module
terraform import module.vpc.aws_subnet.public[0] subnet-0abc123

# Import with for_each (use the key)
terraform import 'aws_instance.servers["web"]' i-0abc123def456789
📝 After import, the resource is in state but your .tf config must match it. Run terraform plan — a clean plan (no changes) confirms success. From Terraform v1.5+, you can also use import blocks directly in config files.
Dangerous terraform force-unlock <lock-id>

Forcefully removes a state lock that was left behind by a crashed or interrupted Terraform operation. A state lock prevents concurrent modifications — when it gets stuck, it blocks all further Terraform runs.

Usage:

# Get the lock ID from the error message, then:
terraform force-unlock 12345678-abcd-1234-abcd-123456789abc

# Skip confirmation prompt
terraform force-unlock -force 12345678-abcd-1234-abcd-123456789abc

Typical Error Message:

Error: Error acquiring the state lock
Lock Info:
  ID:     12345678-abcd-1234-abcd-123456789abc
  Who:    user@hostname
  Created: 2024-01-15 09:32:10
⚠️ DANGER: Only force-unlock if you are 100% certain no other Terraform operation is actively running. Unlocking during an active apply can corrupt state and cause unpredictable infrastructure changes.
Safe terraform apply -refresh-only

Reconciles the state file with real infrastructure by querying the provider for current values — without making any actual changes. This is the modern, safer replacement for the deprecated terraform refresh command.

Usage:

# Preview what has drifted (no changes made)
terraform plan -refresh-only

# Apply the refresh to update state
terraform apply -refresh-only

# Refresh without prompting for confirmation
terraform apply -refresh-only -auto-approve
✅ Production use: Run periodically to detect infrastructure drift — situations where someone made manual changes in the cloud console that Terraform doesn't know about yet.

Quick Reference Summary

Command What It Does Risk Level Modifies State?
state list List all tracked resources Low No
state show Show full details of a resource Low No
state pull Download and print raw state Low No
apply -refresh-only Sync state with real infrastructure Low Yes (safe)
state mv Rename / move resource in state Medium Yes
state rm Remove resource from state (no destroy) Medium Yes
state replace-provider Swap provider source in state Medium Yes
import Bring existing resource into state Medium Yes
state push Upload local state to remote backend High Yes (overwrites)
force-unlock Remove a stuck state lock High Yes
