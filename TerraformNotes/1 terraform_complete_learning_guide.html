What is Terraform?
Understanding Infrastructure as Code (IaC)

Terraform is an open-source Infrastructure as Code (IaC) tool created by HashiCorp. It lets you define, provision, and manage cloud infrastructure using simple configuration files — instead of clicking around in a cloud console manually.

🧠 Think of it this way: Just like a developer writes code to build an application, a DevOps/Cloud engineer writes Terraform code to build infrastructure (servers, databases, networks, etc.). That code can be versioned, reviewed, and reused — exactly like software.

Why Terraform? — The Problem It Solves

Before IaC, teams created infrastructure manually through cloud consoles. This caused:

  • No repeatability — "It works on my account" problems
  • No version history — You couldn't see who changed what, when
  • Slow onboarding — New environments took days to set up
  • Human error — A wrong click could destroy production

Terraform solves all of these. Your entire infrastructure is described in .tf files that live in Git.

Terraform vs Other Tools

ToolTypeBest For
TerraformIaC — ProvisioningCreating and managing cloud resources (multi-cloud)
AWS CloudFormationIaC — AWS onlyAWS-only teams who want native integration
AnsibleConfiguration ManagementConfiguring software INSIDE servers
PulumiIaC — Code-firstTeams that prefer Python/TypeScript over HCL
💡 Key Point: Terraform is cloud-agnostic. The same workflow works for AWS, Azure, GCP, and 1000+ providers. This is its biggest advantage over tools like CloudFormation.

How Terraform Works
The core workflow and key concepts every user must know

The 3-Step Core Workflow

✍️ Write
.tf files
→
📋 Plan
Preview changes
→
🚀 Apply
Create infra
# Step 1: Write your config in .tf files (next sections)

# Step 2: Preview what Terraform WILL do — no changes made yet
terraform plan

# Step 3: Actually create/modify the infrastructure
terraform apply

# To destroy everything managed by this config
terraform destroy

Key Concepts You Must Understand

📄 Configuration Files (.tf)

Files where you write infrastructure code in HCL (HashiCorp Configuration Language). Terraform reads all .tf files in a directory.

🗂️ State File (terraform.tfstate)

Terraform's memory. It tracks what resources it has created. Never delete or edit this manually.

🔌 Providers

Plugins that let Terraform talk to cloud platforms (AWS, Azure, GCP). Think of them as drivers for your cloud.

📦 Resources

The actual infrastructure objects Terraform creates — EC2 instances, S3 buckets, VPCs, databases, etc.

How Terraform Knows What to Change

Terraform compares 3 things every time you run plan:

📄 Your Code
desired state
vs
🗂️ State File
last known state
vs
☁️ Real Cloud
actual state

If they differ, Terraform shows you a plan of exactly what it will create, modify, or destroy.

Installation & Setup
Get Terraform running on your machine

Install Terraform

# macOS (via Homebrew)
brew tap hashicorp/tap
brew install hashicorp/tap/terraform

# Ubuntu / Debian Linux
wget -O- https://apt.releases.hashicorp.com/gpg | sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt update && sudo apt install terraform

# Windows (via Chocolatey)
choco install terraform

# Verify installation
terraform -version
# Output: Terraform v1.6.3

Configure AWS Provider (Most Common)

# Install AWS CLI and configure credentials
aws configure
# Enter: AWS Access Key ID, Secret Key, Region, Output format

# OR use environment variables (recommended for CI/CD)
export AWS_ACCESS_KEY_ID="your-access-key"
export AWS_SECRET_ACCESS_KEY="your-secret-key"
export AWS_DEFAULT_REGION="us-east-1"

Project File Structure

# Recommended file layout for any Terraform project
my-project/
├── main.tf          # Core resources go here
├── variables.tf     # Input variable declarations
├── outputs.tf       # Output value declarations
├── versions.tf      # Terraform & provider version requirements
├── terraform.tfvars # Actual variable values (don't commit secrets!)
└── .gitignore       # Exclude .tfstate, .tfvars with secrets

Essential CLI Commands for Beginners

CommandWhat It DoesWhen to Use
terraform initDownloads providers and sets up backendAlways run first, or after adding a new provider
terraform fmtAuto-formats your .tf filesBefore committing code
terraform validateChecks for syntax errorsAfter writing new code
terraform planShows what WILL happen — no changes yetBefore every apply
terraform applyCreates/updates infrastructureWhen plan looks correct
terraform destroyDestroys all managed resourcesCleanup dev environments
⚠️ Golden Rule: ALWAYS run terraform plan before terraform apply. Read the plan output carefully. Never blindly apply — especially in production.

Your First Resource
Write and deploy real infrastructure step by step

Anatomy of a Resource Block

# resource "PROVIDER_TYPE" "LOCAL_NAME" { ... }
#          ↑ what to create     ↑ your name for it (used in references)

resource "aws_instance" "my_server" {
  ami           = "ami-0c55b159cbfafe1f0"  # Amazon Machine Image ID
  instance_type = "t2.micro"              # Size of the server

  tags = {
    Name = "MyFirstServer"
  }
}

Step-by-Step: Create an EC2 Instance

Step 1 — versions.tf
terraform {
  required_version = "~> 1.6"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
Step 2 — main.tf
# Configure the provider
provider "aws" {
  region = "us-east-1"
}

# Create a Security Group
resource "aws_security_group" "web_sg" {
  name        = "web-security-group"
  description = "Allow HTTP and SSH"

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# Create the EC2 Instance
resource "aws_instance" "web_server" {
  ami                    = "ami-0c55b159cbfafe1f0"
  instance_type          = "t2.micro"
  vpc_security_group_ids = [aws_security_group.web_sg.id]  # Reference SG above

  tags = {
    Name = "WebServer"
  }
}
Step 3 — Run it
# Initialize (downloads AWS provider)
terraform init

# Check for errors
terraform validate

# Preview — Terraform will show: Plan: 2 to add
terraform plan

# Create the resources
terraform apply
# Type 'yes' when prompted
💡 Notice: The EC2 instance references aws_security_group.web_sg.id — Terraform automatically detects this dependency and creates the security group FIRST. This is called an implicit dependency.

Variables & Outputs
Make your code dynamic, reusable, and configurable

Hard-coding values like AMI IDs and instance types is fine for learning, but in real projects you need variables so the same code works for dev, staging, and prod without any changes.

Input Variables

# variables.tf
variable "instance_type" {
  type        = string
  description = "EC2 instance type"
  default     = "t2.micro"  # Optional default value
}

variable "environment" {
  type        = string
  description = "Deployment environment (dev/staging/prod)"
  # No default = REQUIRED — Terraform will ask for it
}

variable "allowed_ports" {
  type    = list(number)
  default = [80, 443, 22]
}

variable "tags" {
  type    = map(string)
  default = {
    Team    = "DevOps"
    Project = "MyApp"
  }
}
# main.tf — use variables with var.
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = var.instance_type     # ← using variable

  tags = merge(var.tags, {
    Name        = "${var.environment}-web-server"  # ← string interpolation
    Environment = var.environment
  })
}

Passing Variable Values

# Option 1: terraform.tfvars file (auto-loaded)
instance_type = "t3.medium"
environment   = "dev"

# Option 2: Named var file
terraform apply -var-file="dev.tfvars"

# Option 3: Command line flag
terraform apply -var="environment=dev" -var="instance_type=t3.micro"

# Option 4: Environment variable (prefix with TF_VAR_)
export TF_VAR_environment="dev"
terraform apply

Variable Validation (Best Practice)

variable "environment" {
  type = string

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}

Sensitive Variables — Never Log Secrets

variable "db_password" {
  type      = string
  sensitive = true  # Terraform hides this in all plan/apply output
}

Output Values

Outputs display useful information after apply — like the IP address of your new server. They can also be read by other Terraform configurations.

# outputs.tf
output "server_public_ip" {
  description = "Public IP of the web server"
  value       = aws_instance.web.public_ip
}

output "server_id" {
  value = aws_instance.web.id
}

output "db_password" {
  value     = var.db_password
  sensitive = true  # Hide in output logs
}
# View outputs after apply
terraform output
terraform output server_public_ip

Locals — Computed/Derived Values

locals {
  name_prefix  = "${var.project}-${var.environment}"
  is_prod      = var.environment == "prod"
  replica_count = local.is_prod ? 3 : 1

  common_tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
  }
}

resource "aws_instance" "app" {
  instance_type = local.is_prod ? "t3.large" : "t3.micro"

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-app"
  })
}

Data Sources
Read existing infrastructure without managing it

Data sources let Terraform fetch information from your cloud provider or existing infrastructure — without creating or managing those resources. Use them to reference things that already exist.

🧠 Analogy: A resource block is like building a new house. A data block is like looking up the address of a house that already exists so you can reference it.
# Always fetch the LATEST Amazon Linux AMI — no more hard-coded AMI IDs!
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }
}

resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id  # ← always latest
  instance_type = "t3.micro"
}
# Fetch an existing VPC (not managed by this config)
data "aws_vpc" "main" {
  filter {
    name   = "tag:Name"
    values = ["production-vpc"]
  }
}

# Use it to place resources inside that VPC
resource "aws_subnet" "app" {
  vpc_id     = data.aws_vpc.main.id
  cidr_block = "10.0.5.0/24"
}
# Fetch current AWS account and region info
data "aws_caller_identity" "current" {}
data "aws_region" "current" {}

output "account_id" { value = data.aws_caller_identity.current.account_id }
output "region"     { value = data.aws_region.current.name }
# Fetch a secret from AWS Secrets Manager
data "aws_secretsmanager_secret_version" "db" {
  secret_id = "prod/myapp/db-password"
}

# Use the secret value — never hard-code passwords!
resource "aws_db_instance" "main" {
  password = data.aws_secretsmanager_secret_version.db.secret_string
}

Meta-Arguments
Special built-in arguments that control resource behavior

Meta-arguments are special arguments available on every resource — regardless of provider. They control HOW a resource behaves, not what it creates.

depends_on — Explicit Dependencies

Terraform normally figures out dependencies automatically. Use depends_on only when it can't — e.g., a hidden dependency not visible in code.

resource "aws_s3_bucket" "app_data" {
  bucket = "my-app-data"
}

resource "aws_instance" "app" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = "t3.micro"

  # App needs S3 to exist but doesn't reference it directly
  depends_on = [aws_s3_bucket.app_data]
}

count — Create Multiple Identical Resources

# Create 3 EC2 instances
resource "aws_instance" "workers" {
  count         = 3
  ami           = data.aws_ami.amazon_linux.id
  instance_type = "t3.micro"

  tags = {
    Name = "Worker-${count.index + 1}"  # Worker-1, Worker-2, Worker-3
  }
}

# Access by index
output "worker_ips" {
  value = aws_instance.workers[*].public_ip  # All IPs
}

for_each — Create Resources from a Map or Set

There is a dedicated Section for Each with all it's use case and common Confuction

Better than count when each resource has a unique identity. Adding/removing an item only affects that specific resource — not others.

# Create S3 buckets for each environment
resource "aws_s3_bucket" "env_buckets" {
  for_each = toset(["dev", "staging", "prod"])
  bucket   = "myapp-${each.key}-data"
}

# Create servers with different sizes using a map
resource "aws_instance" "servers" {
  for_each = {
    web = "t3.small"
    app = "t3.medium"
    db  = "t3.large"
  }

  ami           = data.aws_ami.amazon_linux.id
  instance_type = each.value   # t3.small / t3.medium / t3.large

  tags = { Name = each.key }   # web / app / db
}

# Access a specific instance
output "web_ip" { value = aws_instance.servers["web"].public_ip }

provider — Select Alternate Provider

# Deploy to two AWS regions simultaneously
provider "aws" { region = "us-east-1" }
provider "aws" { alias = "west"; region = "us-west-2" }

resource "aws_instance" "east" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"
  # Uses default provider (us-east-1)
}

resource "aws_instance" "west" {
  provider      = aws.west             # ← explicitly use alias
  ami           = "ami-09b4b74c"
  instance_type = "t3.micro"
}

lifecycle — Control Create/Update/Destroy

# Zero-downtime replacement
resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = "t3.medium"

  lifecycle {
    create_before_destroy = true   # New instance created BEFORE old is destroyed
  }
}

# Protect critical database from accidental destruction
resource "aws_db_instance" "prod" {
  identifier     = "prod-db"
  engine         = "postgres"
  instance_class = "db.r5.large"

  lifecycle {
    prevent_destroy = true         # terraform destroy will throw an ERROR
    ignore_changes  = [password]   # Ignore password rotations done externally
  }
}

# force replacement when a dependency changes (Terraform v1.2+)
resource "aws_autoscaling_group" "asg" {
  lifecycle {
    replace_triggered_by = [aws_launch_template.app]
  }
}

Provisioners
Run scripts during resource creation or destruction

Provisioners execute scripts either locally on your machine or remotely on a created resource. They're used to bootstrap software after a server is created.

⚠️ Important: Provisioners are a last resort. If a provisioner fails, Terraform marks the resource as tainted (to be destroyed and recreated on next apply). Always prefer user_data for EC2 bootstrapping when possible.

remote-exec — Run Commands ON the Server

resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = "t3.micro"
  key_name      = "my-keypair"

  # SSH connection details
  connection {
    type        = "ssh"
    user        = "ec2-user"
    private_key = file("~/.ssh/id_rsa")
    host        = self.public_ip
  }

  # Commands to run on the server after it starts
  provisioner "remote-exec" {
    inline = [
      "sudo yum update -y",
      "sudo yum install -y nginx",
      "sudo systemctl start nginx",
    ]
  }
}

local-exec — Run Commands on YOUR Machine

resource "aws_instance" "app" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = "t3.micro"

  # Runs LOCALLY after the instance is created
  provisioner "local-exec" {
    command = "echo 'Server created: ${self.public_ip}' >> servers.txt"
  }

  # Trigger Ansible after creation
  provisioner "local-exec" {
    command = "ansible-playbook -i '${self.public_ip},' playbook.yml"
  }

  # Run on DESTROY
  provisioner "local-exec" {
    when    = destroy
    command = "echo 'Server ${self.id} destroyed'"
  }
}

user_data — Preferred Alternative for EC2

# Better than remote-exec for EC2 bootstrapping
# Runs automatically on first boot — no SSH needed
resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = "t3.micro"

  user_data = base64encode(<<-EOF
    #!/bin/bash
    yum update -y
    yum install -y nginx
    systemctl start nginx
    systemctl enable nginx
  EOF)
}

Workspaces
Manage multiple environments from one configuration

Workspaces let you create multiple separate state files from a single Terraform configuration. Perfect for managing dev, staging, and prod without duplicating code.

🧠 Think of workspaces like Git branches — same codebase, different isolated states. Your code stays the same, but dev and prod have their own independent resources.

Workspace Commands

# List all workspaces (* marks current)
terraform workspace list
# * default
#   dev
#   staging
#   prod

# Create a new workspace
terraform workspace new dev
terraform workspace new staging
terraform workspace new prod

# Switch to a workspace
terraform workspace select prod

# Show current workspace
terraform workspace show

Use Workspace Name in Configuration

# main.tf — different config per workspace
locals {
  workspace_config = {
    dev = {
      instance_type = "t3.micro"
      instance_count = 1
    }
    staging = {
      instance_type = "t3.small"
      instance_count = 2
    }
    prod = {
      instance_type = "t3.large"
      instance_count = 5
    }
  }

  # terraform.workspace = "dev", "staging", or "prod"
  config = local.workspace_config[terraform.workspace]
}

resource "aws_instance" "app" {
  count         = local.config.instance_count
  ami           = data.aws_ami.amazon_linux.id
  instance_type = local.config.instance_type

  tags = {
    Name        = "${terraform.workspace}-app-${count.index + 1}"
    Environment = terraform.workspace
  }
}
# Deploy to dev
terraform workspace select dev
terraform apply

# Deploy to prod (completely separate state)
terraform workspace select prod
terraform apply
⚠️ Limitation: Workspaces share the same backend and provider credentials. For strong isolation (different AWS accounts per environment), use separate root modules or separate backend configurations instead.

Modules
Package and reuse Terraform configurations across projects

A module is simply a folder containing .tf files. You've already been writing modules — your project directory is the root module. Child modules are reusable components you call from the root.

🧠 Think of modules like functions in programming. Instead of copy-pasting the same EC2 + security group code into 10 projects, you write it once as a module and call it everywhere. Change it once → all projects benefit.

Creating Your First Module

# Folder structure
project/
├── main.tf               # Root module — calls child modules
├── variables.tf
├── outputs.tf
└── modules/
    └── web-server/       # ← This is your module
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
# modules/web-server/variables.tf
variable "instance_type" { type = string; default = "t3.micro" }
variable "environment"   { type = string }
variable "name"          { type = string }
# modules/web-server/main.tf
resource "aws_instance" "this" {
  ami           = data.aws_ami.amazon_linux.id
  instance_type = var.instance_type

  tags = {
    Name        = var.name
    Environment = var.environment
  }
}

data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]
  filter { name = "name"; values = ["amzn2-ami-hvm-*-x86_64-gp2"] }
}
# modules/web-server/outputs.tf
output "instance_id" { value = aws_instance.this.id }
output "public_ip"   { value = aws_instance.this.public_ip }
# Root main.tf — calling the module
module "dev_server" {
  source        = "./modules/web-server"
  environment   = "dev"
  name          = "dev-web-01"
  instance_type = "t3.micro"
}

module "prod_server" {
  source        = "./modules/web-server"
  environment   = "prod"
  name          = "prod-web-01"
  instance_type = "t3.large"
}

# Access module outputs
output "prod_server_ip" {
  value = module.prod_server.public_ip
}
📝 After adding or updating a module source, always run terraform init again to download/update it.

Using Public Registry Modules PROD

The Terraform Registry has thousands of pre-built, community-maintained modules. Don't reinvent the wheel.

# Use the official AWS VPC module — saves hundreds of lines
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.1.2"  # ALWAYS pin the version!

  name = "my-vpc"
  cidr = "10.0.0.0/16"
  azs             = ["us-east-1a", "us-east-1b"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24"]
  enable_nat_gateway = true
}

# Pass VPC outputs into another module
module "web_server" {
  source      = "./modules/web-server"
  environment = "prod"
  name        = "prod-web"
  subnet_id   = module.vpc.private_subnets[0]  # ← chain modules together
}

Backends & Remote State
Store state safely, enable team collaboration, share outputs

By default Terraform saves state locally in terraform.tfstate. This is fine when learning, but in teams this causes conflicts. Remote backends store state in a shared, locked, encrypted location.

⚠️ Never use local state in production. If your laptop dies, your state is gone. If two engineers run apply at the same time, state corruption can occur.

S3 Backend with DynamoDB Locking PROD

# backend.tf
terraform {
  backend "s3" {
    bucket         = "mycompany-terraform-state"  # S3 bucket stores the state
    key            = "prod/webapp/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true                          # Encrypt state at rest
    dynamodb_table = "terraform-state-lock"       # Prevents simultaneous applies
  }
}
# One-time setup: create the S3 bucket and DynamoDB table
resource "aws_s3_bucket" "state" {
  bucket = "mycompany-terraform-state"
  lifecycle { prevent_destroy = true }   # Never delete this!
}

resource "aws_s3_bucket_versioning" "state" {
  bucket = aws_s3_bucket.state.id
  versioning_configuration { status = "Enabled" }  # State history
}

resource "aws_dynamodb_table" "lock" {
  name         = "terraform-state-lock"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"
  attribute { name = "LockID"; type = "S" }
}

Sharing State Between Teams PROD

Large organizations split infrastructure into separate state files. Teams share outputs via terraform_remote_state.

# networking team owns VPC → outputs vpc_id
output "vpc_id" { value = aws_vpc.main.id }

# app team reads networking state to get vpc_id
data "terraform_remote_state" "networking" {
  backend = "s3"
  config = {
    bucket = "mycompany-terraform-state"
    key    = "prod/networking/terraform.tfstate"
    region = "us-east-1"
  }
}

resource "aws_instance" "app" {
  subnet_id = data.terraform_remote_state.networking.outputs.private_subnet_id
}
💡 Key Convention: Structure your S3 state keys by environment and component:
{env}/{team}/{component}/terraform.tfstate
Example: prod/platform/networking/terraform.tfstate

State Commands
Inspect, move, import and repair Terraform state

State commands are how you inspect and surgically modify what Terraform is tracking. These are essential operational skills for any production engineer.

CommandPurposeProduction Use Case
terraform state listList all tracked resourcesAudit before any state operation
terraform state show <res>Inspect resource attributesDebug without going to AWS console
terraform state mvMove/rename resource in stateAfter refactoring code into modules
terraform state rmRemove resource from stateHand off resource to another team
terraform state pullDownload raw state to stdoutManual backup before risky operations
terraform state pushUpload state to backendRestore corrupted remote state
terraform importImport existing infra into stateAdopt manually created resources
terraform force-unlockRemove stuck state lockAfter a crashed CI/CD pipeline
terraform apply -refresh-onlySync state with real infraAfter manual changes in AWS console

Real Workflow: Moving Resource into a Module

# Scenario: You wrote aws_instance.web at root level.
# You refactor it into module "web". Without state mv,
# Terraform will DESTROY the old one and CREATE a new one!

# 1. See current state
terraform state list
# aws_instance.web   ← old address

# 2. Move state to match new code structure
terraform state mv aws_instance.web module.web.aws_instance.this

# 3. Verify
terraform state list
# module.web.aws_instance.this   ← new address

# 4. Plan should now show: No changes.
terraform plan

Importing Manually Created Resources

# Scenario: Someone manually created an S3 bucket in the console.
# You want Terraform to manage it going forward.

# 1. Write the resource block in your .tf file first
resource "aws_s3_bucket" "legacy" {
  bucket = "my-old-manual-bucket"
}

# 2. Import the real resource into state
terraform import aws_s3_bucket.legacy my-old-manual-bucket

# 3. Run plan to see if any attributes need updating
terraform plan

Best Practices
Patterns used by mature infrastructure teams worldwide

📌 Pin All Versions

Pin provider and module versions. Use ~> 5.0 (allows patch, not major updates). Never leave versions unpinned in production.

🔒 Never Store Secrets in Code

Use AWS Secrets Manager, SSM Parameter Store, or HashiCorp Vault. Fetch secrets with data sources at apply time.

🏷️ Tag Everything

Apply consistent tags: Environment, Team, Project, ManagedBy=terraform. Use a locals block for common tags.

📁 One Root per Environment

Use separate directories (or repos) for dev/staging/prod. Avoids accidental cross-environment applies.

✅ Always Plan Before Apply

Run terraform plan -out=tfplan and read it carefully. In CI/CD, require human approval for prod.

🔄 Keep State Backed Up

Enable S3 versioning on your state bucket. You can roll back to previous state versions if something goes wrong.

versions.tf — Always Include This File

# versions.tf — lock your toolchain
terraform {
  required_version = "~> 1.6"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

Standard Tagging with Locals

locals {
  common_tags = {
    Environment = var.environment
    Project     = var.project
    Team        = var.team
    ManagedBy   = "terraform"
  }
}

# Apply to every resource
resource "aws_instance" "app" {
  tags = merge(local.common_tags, { Name = "app-server" })
}

CI/CD Integration
Automate Terraform with GitHub Actions — the production standard

In production, Terraform is never run manually by engineers. Every change goes through a CI/CD pipeline that validates, plans, and applies changes with proper review gates.

📝 PR Opened
→
✅ fmt + validate
→
📋 Plan (comment on PR)
→
👤 Human Review
→
🔀 Merge to main
→
🚀 Auto Apply

GitHub Actions Pipeline PROD

# .github/workflows/terraform.yml
name: Terraform CI/CD

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  terraform:
    runs-on: ubuntu-latest
    environment: production

    steps:
      - uses: actions/checkout@v4

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: "1.6.3"

      # Authenticate with AWS using OIDC (no long-lived keys!)
      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/TerraformCIRole
          aws-region: us-east-1

      - name: Terraform Init
        run: terraform init

      - name: Terraform Format Check
        run: terraform fmt -check -recursive

      - name: Terraform Validate
        run: terraform validate

      - name: Terraform Plan
        run: terraform plan -out=tfplan -var-file=environments/prod.tfvars
        env:
          TF_VAR_db_password: ${{ secrets.DB_PASSWORD }}

      # Only apply when code is merged to main
      - name: Terraform Apply
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        run: terraform apply tfplan

Key CI/CD CLI Flags

# Save plan to file (apply exactly what was planned)
terraform plan  -out=tfplan
terraform apply tfplan

# Non-interactive apply (no "yes" prompt) — for CI only
terraform apply -auto-approve

# Exit codes for scripting:
# 0 = no changes  |  1 = error  |  2 = changes present
terraform plan -detailed-exitcode

# Target a single resource (use sparingly)
terraform apply -target=module.vpc
💡 Security Tip: Use AWS OIDC with GitHub Actions instead of long-lived access keys. Create an IAM role that trusts GitHub's OIDC provider — credentials are temporary and automatically rotated.
