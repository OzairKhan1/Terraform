Section 01

What is the Terraform State File?

Terraform uses a file called terraform.tfstate to keep track of all real infrastructure it has created. It maps your code to actual cloud resources.

What's inside tfstate?

Resource IDs, IP addresses, ARNs, secrets, passwords, connection strings — essentially a complete snapshot of your infrastructure in JSON format. This is why it must be secured.

terraform.tfstate (simplified)
{
  "version": 4,
  "resources": [
    {
      "type": "aws_instance",
      "name": "web",
      "instances": [{
        "attributes": {
          "id":         "i-0abc123def456",
          "public_ip":  "54.12.34.56",
          "private_key": "-----BEGIN RSA..."  ← sensitive!
        }
      }]
    }
  ]
}
Section 02

The Problem with Local State

By default Terraform saves state locally as terraform.tfstate. This causes serious problems in team environments:

ProblemWhat HappensRisk
No sharing Other team members can't see current state Duplicate resources created
No locking Two people run terraform apply simultaneously State corruption / resource conflicts
No versioning State file gets overwritten with no history Can't roll back after bad apply
No encryption Secrets sit in plaintext on dev laptop Security breach if laptop is lost
Accidental commit tfstate pushed to Git repo Credentials exposed publicly
⚠ Critical Rule

NEVER commit terraform.tfstate to Git. Add it to .gitignore immediately. It contains plaintext secrets.

Section 03

S3 + DynamoDB Architecture

The industry-standard solution is a remote backend using S3 for storage and DynamoDB for state locking.

👨‍💻 Developer A
terraform apply
→
🔒 DynamoDB Lock
acquires lock first
→
🪣 S3 Bucket
reads/writes state
👩‍💻 Developer B
terraform apply
→
🚫 Lock Blocked
waits — lock held by A

S3 stores the state file securely with encryption and versioning. DynamoDB acts as a distributed lock — only one person can modify state at a time.

Why this combination?

S3 alone can't prevent simultaneous writes. DynamoDB alone can't store state. Together they solve both problems: secure storage + safe concurrency.

Section 04

Step-by-Step Setup

1

Create S3 Bucket for State Storage

bootstrap/s3.tf
resource "aws_s3_bucket" "tf_state" {
  bucket = "mycompany-terraform-state-prod"   # must be globally unique

  # Prevent accidental deletion of this critical bucket
  lifecycle {
    prevent_destroy = true
  }
}

# Enable versioning — recover previous state anytime
resource "aws_s3_bucket_versioning" "tf_state" {
  bucket = aws_s3_bucket.tf_state.id
  versioning_configuration {
    status = "Enabled"
  }
}

# Server-side encryption — state always encrypted at rest
resource "aws_s3_bucket_server_side_encryption_configuration" "tf_state" {
  bucket = aws_s3_bucket.tf_state.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"   # or "aws:kms" for KMS key
    }
  }
}

# Block ALL public access — state must never be public
resource "aws_s3_bucket_public_access_block" "tf_state" {
  bucket                  = aws_s3_bucket.tf_state.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}
2

Create DynamoDB Table for State Locking

bootstrap/dynamodb.tf
resource "aws_dynamodb_table" "tf_lock" {
  name         = "terraform-state-lock"
  billing_mode = "PAY_PER_REQUEST"   # no capacity planning needed

  # MUST be named exactly "LockID" — Terraform requires this
  hash_key = "LockID"

  attribute {
    name = "LockID"
    type = "S"   # S = String type
  }

  tags = {
    Name    = "Terraform State Lock"
    Purpose = "terraform-locking"
  }
}
⚠ Important

The DynamoDB partition key must be named LockID exactly. Terraform looks for this specific key name — it won't work otherwise.

3

Configure the Backend in Your Project

backend.tf
terraform {
  backend "s3" {
    bucket         = "mycompany-terraform-state-prod"
    key            = "prod/ec2/terraform.tfstate"   # path inside bucket
    region         = "us-east-1"
    encrypt        = true                            # encrypt in transit too

    # DynamoDB locking
    dynamodb_table = "terraform-state-lock"
  }
}
💡 Key naming convention

Use a structured path for key: environment/service/terraform.tfstate. Example: prod/networking/terraform.tfstate, dev/ec2/terraform.tfstate. This keeps multiple projects in one bucket organized.

4

Initialize and Migrate State

terminal
# Run after adding backend config
$ terraform init

# Terraform will ask: "Do you want to copy existing state to the new backend?"
> yes

# Verify state is now in S3
$ terraform state list

# Local tfstate is now safe to delete (state is in S3)
$ rm terraform.tfstate terraform.tfstate.backup
Section 05

Complete Production Backend Config

Here is a full production-ready setup with KMS encryption and all recommended options:

backend.tf — production grade
terraform {
  required_version = ">= 1.5.0"

  backend "s3" {
    # Storage
    bucket  = "mycompany-terraform-state"
    key     = "${var.environment}/terraform.tfstate"
    region  = "us-east-1"

    # Encryption
    encrypt        = true
    kms_key_id     = "arn:aws:kms:us-east-1:123456789:key/abc-123"   # KMS key for extra security

    # Locking
    dynamodb_table = "terraform-state-lock"

    # IAM role (for CI/CD pipelines)
    role_arn       = "arn:aws:iam::123456789:role/TerraformStateRole"
  }
}

# terraform.tfstate.backup

Section 06

Security Best Practices

S3 Bucket Hardening

  • Enable S3 versioning — roll back to any previous state version
  • Enable AES-256 or KMS encryption at rest
  • Block all public access (4 settings)
  • Enable S3 access logging to audit who accessed state
  • Use prevent_destroy = true lifecycle rule on the bucket
  • Enable MFA delete — prevents accidental state deletion

DynamoDB Table Hardening

  • Use PAY_PER_REQUEST billing — no over-provisioning needed
  • Enable DynamoDB encryption at rest
  • Add CloudWatch alarms for unusual lock activity

IAM Permissions — Least Privilege

iam-policy.json — minimal permissions
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject"
      ],
      "Resource": "arn:aws:s3:::mycompany-terraform-state/*"
    },
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",      ← check if lock exists
        "dynamodb:PutItem",      ← acquire lock
        "dynamodb:DeleteItem"    ← release lock
      ],
      "Resource": "arn:aws:dynamodb:*:*:table/terraform-state-lock"
    }
  ]
}

CI/CD Pipeline Best Practice

  • Use IAM roles for CI/CD — never hardcode AWS credentials
  • Use separate state files per environment (dev/staging/prod)
  • Add plan step before apply — require approval on prod
  • Never store .terraform/ directory in artifacts
Section 07

Multi-Environment State with Workspaces

Terraform workspaces let you manage multiple environment states from the same config using one backend.

terminal — workspace commands
# Create workspaces for each environment
$ terraform workspace new dev
$ terraform workspace new staging
$ terraform workspace new prod

# Switch between workspaces
$ terraform workspace select prod

# See current workspace
$ terraform workspace show
prod

# Each workspace gets its own state path in S3:
# env:/dev/terraform.tfstate
# env:/staging/terraform.tfstate
# env:/prod/terraform.tfstate
main.tf — using workspace in resources
locals {
  instance_type = {
    dev     = "t2.micro"
    staging = "t2.medium"
    prod    = "t3.large"
  }
}

resource "aws_instance" "web" {
  instance_type = local.instance_type[terraform.workspace]  # auto picks by workspace
  ami           = "ami-0c55b159cbfafe1f0"
}
Industry note

Many large teams prefer separate backend configs per environment over workspaces — it provides stronger isolation. Workspaces share the same backend config which can be a risk. Use workspaces for simpler setups, separate backends for strict isolation.

Section 08 · Interview Prep

Interview Questions — Terraform State

BASIC

What is the Terraform state file and why is it important?

The state file (terraform.tfstate) is Terraform's source of truth. It maps your configuration code to real infrastructure resources in the cloud. Without it, Terraform can't know what already exists — it would try to recreate everything on every apply.

It tracks resource IDs, metadata, dependencies, and outputs. It also contains sensitive data like passwords and private keys, which is why securing it is critical.

BASIC

What is a remote backend and why do we use it?

A remote backend stores the state file in a shared, centralized location (like S3) instead of on a local developer machine. This enables teams to collaborate safely — everyone reads and writes the same state.

It also enables state locking, versioning, encryption, and access control — none of which are possible with local state.

MEDIUM

What is state locking and what happens if it fails?

State locking prevents two operations from modifying state simultaneously. When you run terraform apply, Terraform writes a lock entry to DynamoDB. Any other apply attempted while the lock is held will immediately fail with a lock error showing who holds it.

If a process crashes mid-apply, the lock may remain. You can force-unlock it using terraform force-unlock <LOCK_ID> — but only after confirming no apply is actually running, as force-unlocking during an active apply can corrupt state.

MEDIUM

Why DynamoDB specifically for state locking? Why not just S3?

S3 doesn't natively support atomic locks. Two concurrent writes to the same S3 object is a race condition — the last write wins and you'd have state corruption. DynamoDB provides atomic PutItem operations with conditional writes, which is exactly what a distributed lock needs.

DynamoDB's LockID item either exists (locked) or doesn't (unlocked) — and this check-and-set is atomic, making it safe for concurrent access.

HARD

How do you manage Terraform state for multiple environments (dev/staging/prod)?

There are two main approaches used in production:

1. Separate state files with different keys: Use the same S3 backend but different key paths — dev/terraform.tfstate, prod/terraform.tfstate. This is simple and transparent.

2. Terraform Workspaces: Use terraform workspace new prod — Terraform automatically namespaces state per workspace. Good for similar environments, but all share the same backend config which reduces isolation.

For strict production environments, most companies use separate AWS accounts per environment with separate backends — this provides the strongest blast radius isolation.

HARD

What happens if two people accidentally corrupt the state? How do you recover?

This is why S3 versioning is mandatory. Every state change creates a new version in S3. To recover:

1. Open S3 console → navigate to the state file → click "Show versions"

2. Identify the last known good version (before corruption)

3. Download that version and restore it: terraform state push old-state.tfstate

4. Run terraform plan to verify the restored state matches reality before any further applies.

You can also use terraform state pull to inspect current state and terraform state mv / terraform state rm for surgical edits.

MEDIUM

What is terraform_remote_state and when would you use it?

terraform_remote_state is a data source that lets one Terraform project read the outputs of another project's state file. This is used to share information between separately managed codebases — for example, a networking project that creates a VPC can expose the VPC ID as an output, and an EC2 project reads it using terraform_remote_state.

It's the Terraform-native way to avoid hardcoding resource IDs between projects. However, it creates a tight coupling — if the upstream state changes, downstream projects are affected.

HARD

A developer accidentally deleted a resource manually in the AWS console. What happens when terraform plan runs and how do you fix it?

Terraform's state still thinks the resource exists, but the real infrastructure doesn't. On next terraform plan, Terraform will detect the drift and show a plan to recreate the deleted resource.

Options to fix it depend on the situation:

If you want to recreate it: Simply run terraform apply — Terraform will recreate the resource as defined in code.

If the resource shouldn't exist anymore: Remove it from your Terraform code AND remove it from state: terraform state rm resource_type.name. Then apply so the code and state agree.

Best practice: Never manually modify resources managed by Terraform. Use terraform import if you need to bring existing resources under Terraform management.

BASIC

What should be in .gitignore for a Terraform project?

At minimum, always ignore these files in every Terraform project:

terraform.tfstate and terraform.tfstate.backup — contain secrets and infrastructure details.

.terraform/ directory — contains downloaded provider binaries and cached modules (large, regenerated by terraform init).

*.tfvars files — often contain secrets, API keys, passwords. Use *.tfvars.example as a template committed to git instead.

crash.log — Terraform crash logs that may contain sensitive trace data.
