Managing Multiple Environments

A complete step-by-step guide to structuring test, dev, and prod environments the way real production teams do it — with isolated remote state, shared modules, and automated pipelines.
🟢 dev 🟡 test 🔴 prod 🗄️ S3 Remote State 🔒 DynamoDB Locking ⚡ Terraform v1.x
1
The 3 Industrial Approaches
Understand your options before choosing

There are three widely-used patterns for managing multiple environments in Terraform. Each has trade-offs. Knowing all three helps you choose the right one for your team.

🗂️ Workspaces

One codebase. Multiple state files in the same backend, switched by workspace name.

✅ Quick to set up
✅ Less code duplication
❌ Same backend credentials for all envs
❌ No true blast-radius isolation
❌ Easy to accidentally apply to wrong env
❌ Not recommended for prod by HashiCorp
⭐ Industry Standard

📁 Directory-per-Environment

Separate folder per environment. Each has its own backend config, state file, and variables.

✅ True isolation
✅ Different AWS accounts per env
✅ Independent state files
✅ Recommended by HashiCorp & AWS
✅ Most teams use this
❌ Some code duplication (solved by modules)

🔧 Terragrunt

A wrapper tool over Terraform that generates backend configs and eliminates all duplication.

✅ Zero duplication (DRY)
✅ Handles complex dependency graphs
✅ Great for 10+ environments
❌ Extra tool to learn
❌ Overkill for small teams
❌ Debugging is harder
💡 Our recommendation and industry standard = Directory-per-Environment. It gives you true isolation without adding tooling complexity. Modules solve the duplication problem. This is what this guide covers end-to-end.

Why Directory-per-Environment Wins
The isolation model that enterprise teams trust

The Core Principle: Separate Everything

Each environment should be completely independent. If prod breaks, dev keeps working. If someone runs terraform destroy in dev, prod is untouched.

What's IsolatedDevTestProd
State File (.tfstate)dev/terraform.tfstatetest/terraform.tfstateprod/terraform.tfstate
Backend ConfigSeparate S3 keySeparate S3 keySeparate S3 key
Variable Valuesdev.tfvarstest.tfvarsprod.tfvars
AWS Account (ideal)dev-accounttest-accountprod-account
Instance Sizest3.microt3.smallt3.large
Replica Count125+

State Key Naming Convention

Every environment gets its own path inside your S3 bucket. This is how you keep state files completely separate:

S3 Bucket: mycompany-terraform-state
├── dev/
│ └── webapp/terraform.tfstate # dev state
├── test/
│ └── webapp/terraform.tfstate # test state
└── prod/
└── webapp/terraform.tfstate # prod state
⚠️ Critical: Never share a state file between environments. One corrupted state file should never bring down another environment.

The Complete Folder Structure
How to organize your repository — the industry standard layout

This is the structure you should build. Don't worry — we'll create each file step by step.

my-terraform-project/
│
├── modules/ # ← Shared, reusable code (write ONCE)
│ └── webapp/ # Your app module
│ ├── main.tf # All resources (EC2, SG, RDS, etc.)
│ ├── variables.tf # Input variables
│ ├── outputs.tf # Outputs
│ └── versions.tf # Provider requirements
│
├── environments/ # ← Each env is separate
│ ├── dev/ # Dev environment
│ │ ├── main.tf # Calls the module
│ │ ├── backend.tf # Dev remote state config
│ │ ├── variables.tf # Variable declarations
│ │ ├── outputs.tf # Outputs
│ │ ├── versions.tf # Provider + terraform version
│ │ └── terraform.tfvars # Dev values (small, cheap)
│ │
│ ├── test/ # Test environment
│ │ ├── main.tf
│ │ ├── backend.tf # Test remote state config
│ │ ├── variables.tf
│ │ ├── outputs.tf
│ │ ├── versions.tf
│ │ └── terraform.tfvars # Test values (medium)
│ │
│ └── prod/ # Production environment
│ ├── main.tf
│ ├── backend.tf # Prod remote state config
│ ├── variables.tf
│ ├── outputs.tf
│ ├── versions.tf
│ └── terraform.tfvars # Prod values (large, HA)
│
└── README.md
💡 The Pattern: The modules/ folder is where your actual infrastructure logic lives — written ONCE. Each environments/ folder is just a thin wrapper that calls the module with different values. This eliminates duplication while keeping environments fully isolated.

Build the Shared Module
Write your infrastructure code ONCE — reuse it across all environments

The module is where all your real resources live. You write this once and all three environments use the same code — only the values differ.

📄 modules/webapp/versions.tf
terraform {
  required_version = "~> 1.6"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
📄 modules/webapp/variables.tf
# Every value that changes between environments becomes a variable

variable "environment" {
  type        = string
  description = "Environment name (dev/test/prod)"
  validation {
    condition     = contains(["dev", "test", "prod"], var.environment)
    error_message = "Must be dev, test, or prod."
  }
}

variable "instance_type" {
  type        = string
  description = "EC2 instance type"
}

variable "instance_count" {
  type        = number
  description = "Number of app servers"
  default     = 1
}

variable "db_instance_class" {
  type        = string
  description = "RDS instance class"
}

variable "db_multi_az" {
  type        = bool
  description = "Enable Multi-AZ for RDS (true in prod)"
  default     = false
}

variable "db_password" {
  type      = string
  sensitive = true
}

variable "vpc_cidr" {
  type    = string
  default = "10.0.0.0/16"
}

variable "tags" {
  type    = map(string)
  default = {}
}
📄 modules/webapp/main.tf
# Fetch latest Amazon Linux 2 AMI
data "aws_ami" "al2" {
  most_recent = true
  owners      = ["amazon"]
  filter { name = "name";              values = ["amzn2-ami-hvm-*-x86_64-gp2"] }
  filter { name = "virtualization-type"; values = ["hvm"] }
}

# Common tags applied to all resources
locals {
  common_tags = merge(var.tags, {
    Environment = var.environment
    ManagedBy   = "terraform"
  })
  name_prefix = "webapp-${var.environment}"
}

# Security Group
resource "aws_security_group" "app" {
  name        = "${local.name_prefix}-sg"
  description = "App security group"

  ingress {
    from_port   = 80;  to_port = 80
    protocol    = "tcp";  cidr_blocks = ["0.0.0.0/0"]
  }
  egress {
    from_port   = 0;  to_port = 0
    protocol    = "-1";  cidr_blocks = ["0.0.0.0/0"]
  }
  tags = merge(local.common_tags, { Name = "${local.name_prefix}-sg" })
}

# EC2 Instances — count varies per environment
resource "aws_instance" "app" {
  count                  = var.instance_count
  ami                    = data.aws_ami.al2.id
  instance_type          = var.instance_type
  vpc_security_group_ids = [aws_security_group.app.id]

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-server-${count.index + 1}"
  })
}

# RDS Database — size and HA vary per environment
resource "aws_db_instance" "main" {
  identifier        = "${local.name_prefix}-db"
  engine            = "postgres"
  engine_version    = "15"
  instance_class    = var.db_instance_class
  username          = "admin"
  password          = var.db_password
  multi_az          = var.db_multi_az       # false in dev/test, true in prod
  skip_final_snapshot = var.environment != "prod"

  # Only protect prod database
  lifecycle {
    prevent_destroy = false  # Override per env using lifecycle if needed
  }

  tags = merge(local.common_tags, { Name = "${local.name_prefix}-db" })
}
📄 modules/webapp/outputs.tf
output "instance_ids"   { value = aws_instance.app[*].id }
output "instance_ips"   { value = aws_instance.app[*].public_ip }
output "db_endpoint"    { value = aws_db_instance.main.endpoint }
output "sg_id"          { value = aws_security_group.app.id }
✅ Module is done. You will NEVER touch this folder again when deploying to different environments. Each environment just passes in different values.
5
Set Up the Remote Backend
Create the S3 bucket and DynamoDB table that will hold all state files

This is a one-time setup. You create one S3 bucket that holds state for all environments — separated by key path. One DynamoDB table handles locking for all environments.

⚠️ This backend infrastructure itself should be created separately — NOT managed by the same Terraform code it stores state for. Create it manually or with a dedicated "bootstrap" Terraform config.
📄 bootstrap/main.tf — Run this once to create the backend
provider "aws" { region = "us-east-1" }

# ─── S3 Bucket for all Terraform state ───────────────────────────
resource "aws_s3_bucket" "tf_state" {
  bucket = "mycompany-terraform-state"

  lifecycle { prevent_destroy = true }   # NEVER accidentally delete
}

resource "aws_s3_bucket_versioning" "tf_state" {
  bucket = aws_s3_bucket.tf_state.id
  versioning_configuration { status = "Enabled" }  # Keep all state history
}

resource "aws_s3_bucket_server_side_encryption_configuration" "tf_state" {
  bucket = aws_s3_bucket.tf_state.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

resource "aws_s3_bucket_public_access_block" "tf_state" {
  bucket                  = aws_s3_bucket.tf_state.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# ─── DynamoDB Table for state locking ────────────────────────────
resource "aws_dynamodb_table" "tf_lock" {
  name         = "terraform-state-lock"
  billing_mode = "PAY_PER_REQUEST"   # No capacity planning needed
  hash_key     = "LockID"

  attribute { name = "LockID"; type = "S" }

  tags = { Name = "Terraform State Lock"; ManagedBy = "terraform" }
}
# Run once to create the backend infrastructure
cd bootstrap/
terraform init
terraform apply

After this runs, your S3 bucket exists and is ready to store state files for dev, test, and prod — in separate paths.

📋 S3 bucket path convention used in this guide:

mycompany-terraform-state/dev/webapp/terraform.tfstate
mycompany-terraform-state/test/webapp/terraform.tfstate
mycompany-terraform-state/prod/webapp/terraform.tfstate

Deploy Each Environment
Step-by-step commands to initialize and apply each environment

Each environment is deployed independently by cd-ing into its folder and running the standard Terraform workflow.

Deploy Dev DEV

# 1. Go into the dev environment folder
cd environments/dev/

# 2. Set your DB password as env var (never hardcode secrets!)
export TF_VAR_db_password="dev-password-123"

# 3. Initialize — downloads providers, connects to S3 backend
terraform init

# 4. Preview what will be created
terraform plan

# 5. Create dev infrastructure
terraform apply

# ✅ State saved to: s3://mycompany-terraform-state/dev/webapp/terraform.tfstate

Deploy Test TEST

# Completely separate — won't touch dev state at all
cd environments/test/

export TF_VAR_db_password="test-password-456"

terraform init    # Connects to test S3 key
terraform plan
terraform apply

# ✅ State saved to: s3://mycompany-terraform-state/test/webapp/terraform.tfstate

Deploy Prod PROD

# Production — always plan to a file first, then review before applying
cd environments/prod/

export TF_VAR_db_password="prod-super-secret-999"

terraform init

# Save the plan to a file — review it carefully before applying!
terraform plan -out=prod.tfplan

# Only apply after thorough review
terraform apply prod.tfplan

# ✅ State saved to: s3://mycompany-terraform-state/prod/webapp/terraform.tfstate
⚠️ Prod Rule: Never run terraform apply -auto-approve on prod. Always use -out to save the plan and review it before applying. In CI/CD, require a manual approval gate for prod.

State Isolation Strategy
How production teams think about keeping environments truly separate

Level 1 — Same Account, Separate Keys (Minimum)

All environments in one AWS account, separated only by S3 state key. Fine for small teams and early projects.

AWS Account: 111111111111 (single account)
├── Dev resources → state: s3://tf-state/dev/webapp/terraform.tfstate
├── Test resources → state: s3://tf-state/test/webapp/terraform.tfstate
└── Prod resources → state: s3://tf-state/prod/webapp/terraform.tfstate

Level 2 — Separate AWS Accounts ENTERPRISE STANDARD

Each environment lives in a completely separate AWS account. This is what AWS recommends and what large companies do. A mistake in dev literally cannot affect prod — they're different accounts.

AWS Organization
├── Account: 111111111111 (dev-account)
│ └── Dev resources + Dev state bucket
├── Account: 222222222222 (test-account)
│ └── Test resources + Test state bucket
└── Account: 333333333333 (prod-account)
└── Prod resources + Prod state bucket 
# Each environment uses a different AWS profile

# environments/dev/versions.tf
provider "aws" {
  region  = "us-east-1"
  profile = "dev-account"    # ← dev AWS credentials
}

# environments/prod/versions.tf
provider "aws" {
  region  = "us-east-1"
  profile = "prod-account"   # ← prod AWS credentials
  # OR use assume_role for cross-account access
  assume_role {
    role_arn = "arn:aws:iam::333333333333:role/TerraformRole"
  }
}

State Locking Explained

When you run terraform apply, DynamoDB creates a lock entry. If another engineer (or CI/CD job) tries to apply at the same time, they get an error — not silent corruption.

# What happens when two people apply simultaneously:

# Engineer A runs apply → DynamoDB creates lock
# LockID = "mycompany-terraform-state/prod/webapp/terraform.tfstate"

# Engineer B runs apply at same time → gets this error:
# Error: Error locking state: ConditionalCheckFailedException
# Lock Info: ID=abc-123, Who=engineer-a@company.com
# Lock is released automatically when Engineer A finishes

# If apply crashes and lock gets stuck:
terraform force-unlock <lock-id>

Migrate Your Existing Code
How to restructure your current single-env Terraform into the multi-env pattern

You mentioned you already have Terraform code for one environment with a remote backend. Here's exactly how to migrate it to the multi-env structure without destroying anything.

Step 1 — Understand your current state
# See what you currently have
terraform state list

# Pull current backend config
cat backend.tf  # Note down your current bucket, key, region
Step 2 — Create the new folder structure
mkdir -p environments/dev environments/test environments/prod
mkdir -p modules/webapp
Step 3 — Move your existing resources into the module
# Copy your current resource code into the module
cp main.tf      modules/webapp/main.tf
cp variables.tf modules/webapp/variables.tf
cp outputs.tf   modules/webapp/outputs.tf

# Clean up the module — remove backend configs from it
# Modules should NOT have backend blocks
Step 4 — Create the dev environment pointing to EXISTING state
# environments/dev/backend.tf
# Use the EXACT SAME bucket/key/region as your current backend
# This way Terraform finds the existing state — nothing gets recreated
terraform {
  backend "s3" {
    bucket         = "your-existing-bucket"         # ← same as before
    key            = "your-existing-key.tfstate"    # ← same as before
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"
  }
}
Step 5 — Handle the state address change
# Your resources were previously at root level:
# aws_instance.web
# aws_security_group.app

# Now they'll be inside a module:
# module.webapp.aws_instance.web
# module.webapp.aws_security_group.app

# You must move state addresses to match the new structure
cd environments/dev/
terraform init

# Move each resource into the module namespace
terraform state mv aws_instance.web           module.webapp.aws_instance.web
terraform state mv aws_security_group.app     module.webapp.aws_security_group.app
terraform state mv aws_db_instance.main       module.webapp.aws_db_instance.main
# ... repeat for every resource
Step 6 — Verify with plan
# Plan should show: No changes. Infrastructure is up-to-date.
# If it shows creates/destroys, check your state mv commands
terraform plan
Step 7 — Create test and prod from scratch
# Test and Prod are new — their state files don't exist yet
cd environments/test/
terraform init    # Creates new empty state at test/webapp/terraform.tfstate
terraform plan    # Shows everything to be created fresh
terraform apply

cd environments/prod/
terraform init
terraform plan
terraform apply
💡 The key insight for migration: As long as backend.tf in dev points to the SAME S3 key as before, Terraform will find the existing state and terraform plan will show no changes — after you've done the state mv commands.

CI/CD Pipeline per Environment
Automate deployments with proper gates between environments

The golden rule: code promotes through environments — dev → test → prod. Each promotion requires the previous environment to be healthy.

PR → Dev
→
Auto Apply Dev
→
Tests Pass
→
Apply Test
→
👤 Approval
→
Apply Prod
📄 .github/workflows/terraform.yml
name: Terraform Multi-Environment Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  TF_VERSION: "1.6.3"

jobs:

  # ─── DEV ─────────────────────────────────────────────────────────
  dev:
    name: Deploy Dev
    runs-on: ubuntu-latest
    environment: dev
    defaults:
      run:
        working-directory: environments/dev   # ← run in dev folder

    steps:
      - uses: actions/checkout@v4

      - uses: hashicorp/setup-terraform@v3
        with: { terraform_version: "${{ env.TF_VERSION }}" }

      - name: Auth AWS (Dev Account)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: "arn:aws:iam::111111111111:role/TerraformRole"
          aws-region: us-east-1

      - run: terraform init
      - run: terraform validate
      - run: terraform plan  -out=tfplan
        env: { TF_VAR_db_password: "${{ secrets.DEV_DB_PASSWORD }}" }

      - name: Apply Dev
        if: github.ref == 'refs/heads/main'
        run: terraform apply tfplan


  # ─── TEST ────────────────────────────────────────────────────────
  test:
    name: Deploy Test
    runs-on: ubuntu-latest
    environment: test
    needs: [dev]            # ← waits for dev to succeed
    if: github.ref == 'refs/heads/main'
    defaults:
      run:
        working-directory: environments/test

    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
        with: { terraform_version: "${{ env.TF_VERSION }}" }
      - name: Auth AWS (Test Account)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: "arn:aws:iam::222222222222:role/TerraformRole"
          aws-region: us-east-1
      - run: terraform init
      - run: terraform plan -out=tfplan
        env: { TF_VAR_db_password: "${{ secrets.TEST_DB_PASSWORD }}" }
      - run: terraform apply tfplan


  # ─── PROD ────────────────────────────────────────────────────────
  prod:
    name: Deploy Prod
    runs-on: ubuntu-latest
    environment: prod       # ← GitHub environment with required reviewers
    needs: [test]           # ← waits for test to succeed
    if: github.ref == 'refs/heads/main'
    defaults:
      run:
        working-directory: environments/prod

    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
        with: { terraform_version: "${{ env.TF_VERSION }}" }
      - name: Auth AWS (Prod Account)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: "arn:aws:iam::333333333333:role/TerraformRole"
          aws-region: us-east-1
      - run: terraform init
      - run: terraform plan -out=tfplan
        env: { TF_VAR_db_password: "${{ secrets.PROD_DB_PASSWORD }}" }
      - name: Apply Prod   # ← requires manual approval in GitHub settings
        run: terraform apply tfplan
💡 Pro Tip: In GitHub, go to Settings → Environments → prod → Add required reviewers. This means someone must manually click "Approve" before prod applies — even if all jobs succeed automatically.

Common Mistakes to Avoid

❌ Mistake✅ Correct Approach
Running terraform apply from the wrong folderAlways pwd and check which env you're in
Hardcoding passwords in terraform.tfvarsUse export TF_VAR_db_password=... or Secrets Manager
Same S3 key for two environmentsEvery env must have a unique key path in backend.tf
Skipping terraform plan in prodAlways plan first, save with -out, then apply that plan
Editing state file manuallyUse terraform state mv and terraform import commands
Forgetting terraform init after changing backendAlways init when switching environments or changing backend
